# CrimeVista Team Workplan

## Team Structure

### 1. Kamran

Role: Team leader, full-stack architecture, backend, data engineering

Responsibilities:

- overall architecture guidance
- backend service design
- data pipeline and database design
- integration of analytics APIs
- final technical decisions

### 2. Saksham

Role: Backend and data engineering

Responsibilities:

- API implementation
- database schema and queries
- data cleaning and normalization
- orchestration of backend services

### 3. Ankur

Role: AI/ML and data intelligence

Responsibilities:

- hotspot detection logic
- anomaly detection
- predictive risk modeling
- explanation of insights for the demo

### 4. Shivaleela

Role: Frontend, data visualization, docs, and presentation

Responsibilities:

- dashboard UI and user experience
- charts, maps, and visual storytelling
- documentation and presentation materials
- demo flow and narration

## Suggested Workflow

1. Kamran and Saksham define the backend and data schema.
2. Ankur prepares the analytics methods and sample insights.
3. Shivaleela builds the dashboard around the backend API and analytics outputs.
4. The team reviews the MVP together for demo readiness.
5. Each day ends with a short check of what is demo-ready and what is still pending.

## Delivery Checklist

- Data is loaded and queryable.
- API endpoints return the expected structure.
- Dashboard screens are navigable and visually understandable.
- At least one AI-based insight is visible and explained.
- The demo story can be presented in a short, polished walkthrough.

## Parallel Workstreams

### Backend Workstream

- API endpoints
- database structure
- ingestion scripts

### Analytics Workstream

- hotspot logic
- network analysis helpers
- risk/anomaly scoring

### Frontend Workstream

- dashboard layout
- map and chart components
- filter interactions

## Daily Focus

- keep the solution aligned to the problem statement
- avoid feature creep
- prioritize the MVP demo experience
- keep documentation updated as work progresses
- communicate blockers early to avoid rework
